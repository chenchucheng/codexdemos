# 产品需求文档：Prompt 优化助手 - V1.0（可开发版）

---

## 1. 综述 (Overview)

### 1.1 项目背景与核心问题
随着 AI 工具普及，用户越来越依赖 Prompt（提示词）来驱动模型输出高质量结果。但大量用户存在以下痛点：

- 不知道如何写清晰、有效的 Prompt
- 输入过于模糊，导致输出质量不稳定
- 缺乏结构化表达能力，难以控制生成结果

本项目旨在提供一个**极简、无门槛的 Prompt 优化工具**：用户输入原始草稿后，系统自动进行改写、补全和增强，帮助用户更快获得高质量 Prompt，并可继续迭代优化。

### 1.2 产品目标
- **目标用户**：无 Prompt 工程经验的普通用户、轻度 AI 用户
- **核心价值**：降低 Prompt 编写门槛，提高一次生成可用率
- **V1 目标**：实现“打开即用 → 输入即优化 → 可复制结果 → 可继续追问”的闭环

### 1.3 范围定义（Scope）
**In Scope（V1 包含）**
- 单页聊天式 Prompt 优化界面
- 输入草稿并发送
- 调用 LLM 接口返回优化后的 Prompt
- 支持继续追问进行二次优化
- 支持复制优化结果
- 基础错误提示与重试

**Out of Scope（V1 不包含）**
- 登录/注册体系
- Prompt 历史云端保存
- 多模型切换
- 付费系统
- 团队协作与分享空间

### 1.4 核心业务流程 / 用户旅程地图
1. **阶段一：进入与初始化** — 用户打开网页，看到可用聊天界面
2. **阶段二：输入需求（Prompt 草稿）** — 用户输入并发送原始需求
3. **阶段三：AI 对话生成** — 系统返回结构化、可直接使用的优化 Prompt
4. **阶段四：结果消费与调整** — 用户复制结果或继续补充要求进行迭代

## 2. 角色定义与成功指标

### 2.1 用户角色
- **主角色：普通用户（无需登录）**
  - 目标：快速把模糊需求变成高质量 Prompt

### 2.2 业务成功指标（北极星与辅助指标）
- **北极星指标**：优化结果被复制率（Copy Rate）
- **辅助指标**：
  - 首屏可交互时间（TTI）
  - 首次发送成功率
  - API 成功率
  - 平均响应时长
  - 错误后重试成功率
  - 会话内二次优化率（继续追问）

### 2.3 V1 验收门槛（上线阈值）
- 首屏加载 ≤ 2s（良好网络）
- 首次发送成功率 ≥ 95%
- 平均响应时长 ≤ 5s（P50）
- 复制按钮点击率 ≥ 30%（灰度观察）

## 3. 用户故事详述 (User Stories)

### 阶段一：进入与初始化

#### US-01：作为用户，我希望打开页面即可进入可用的 Prompt 优化界面，以便快速开始与 AI 交互

**价值陈述（Value Statement）**
- 作为普通用户（无需登录）
- 我希望打开网页后直接进入聊天界面
- 以便无门槛使用 Prompt 优化能力

**业务规则与逻辑（Business Logic）**
1. 前置条件
   - 用户访问 Web 页面（Vercel 部署）
   - 浏览器支持 JS
2. 操作流程（Happy Path）
   1) 用户打开页面
   2) 系统渲染聊天 UI
   3) 页面展示标题、引导文案、输入框（placeholder）
   4) AI 不主动发言
   5) 用户可直接输入
3. 异常处理（Error Handling）
   - 页面加载失败：提示“页面加载失败，请刷新”
   - JS 异常：展示降级提示页
   - API Key 未配置：首次发送时报错提示
4. 性能与容量
   - 首屏加载 ≤ 2s
   - 资源体积 ≤ 500KB（不含第三方远程资源）
   - 无本地持久化存储

**验收标准（Acceptance Criteria）**
- GIVEN 用户访问页面 WHEN 加载完成 THEN 显示聊天界面与引导文案
- GIVEN 用户未登录 WHEN 打开页面 THEN 可直接使用

**技术实现概要（Technical Brief）**
- 前端：Next.js + React + useState
- 后端：无业务后端（仅 API 路由转发）
- 数据库：无

**US-01 开发计划（含架构考虑）**

1. **架构基线与目录规划（Day 1）**
   - 目标：先约束“最小架构骨架”，避免后续 US-02/03/04 返工。
   - 建议目录：
     - `app/page.tsx`：页面组装层（仅编排，不承载复杂逻辑）
     - `components/chat/*`：UI 组件层（Header、MessageList、Composer）
     - `lib/config.ts`：运行时配置读取（如环境变量校验）
     - `lib/telemetry.ts`：埋点封装（page_view、js_error）
     - `app/error.tsx`：全局错误边界页面
   - 架构原则：
     - **UI 与状态分离**：页面层管理状态，组件层尽量纯展示
     - **配置集中化**：API 可用性检查统一在 `lib/config.ts`
     - **可观测优先**：初始化阶段即接入基础埋点与错误日志

2. **初始化渲染链路实现（Day 1~2）**
   - 实现内容：
     - 渲染标题、引导文案、输入框 placeholder
     - 默认无 AI 首条欢迎语（保持“用户先发起”）
     - 聊天容器布局与响应式（>=360px）
   - 技术要点：
     - 首屏组件避免引入重依赖，减少 JS 体积
     - 静态文案可服务端渲染，降低白屏感知时间

3. **异常与降级体系（Day 2）**
   - 页面加载失败：在 `app/error.tsx` 提供“刷新重试”提示
   - JS 运行异常：通过 Error Boundary 展示降级页
   - API Key 未配置：首次触发发送时给出明确错误文案（与错误码对齐）
   - 说明：US-01 不直接调用模型，但要在架构上预留“配置检查能力”

4. **性能与工程质量控制（Day 2~3）**
   - 性能预算：
     - 首屏加载 ≤ 2s
     - 前端关键资源目标 ≤ 500KB（gzip 后以构建报告为准）
   - 工程措施：
     - 路由级代码最小化，避免在首页引入高体积库
     - 图片/图标按需加载；无必要不引入客户端状态库
     - 使用 Lighthouse / Web Vitals 做基线采样

5. **验收与发布准备（Day 3）**
   - 手工验收清单：
     - 未登录直接可进入聊天页
     - 标题、引导文案、输入框展示正确
     - AI 不主动发言
     - 异常时有可理解提示
   - 发布前检查：
     - Vercel 环境变量命名与读取逻辑已校验
     - 埋点 `page_view` 与 `js_error` 可观测

6. **US-01 与后续故事的架构衔接**
   - 对 US-02：复用 Composer 与基础状态容器，最小改动接入发送逻辑
   - 对 US-03：在现有消息流中接入 `assistant` 消息渲染即可扩展
   - 对 US-04：在消息组件层扩展复制按钮，避免重写页面结构
   - 结论：US-01 阶段先做“可扩展骨架”，确保后续三阶段以增量方式开发

**US-01 里程碑定义（DoD）**
- DoD-1：页面可访问且首屏功能元素完整
- DoD-2：异常/降级路径可触发且提示正确
- DoD-3：关键性能指标达到门槛
- DoD-4：埋点与日志可用于上线后排障

### 阶段二：输入需求（Prompt 草稿）

#### US-02：作为用户，我希望输入并发送 Prompt 需求，以便让 AI 生成优化后的提示词

**价值陈述**
- 作为用户
- 我希望输入文本并发送
- 以便获取 AI 优化结果

**业务规则与逻辑**
1. 前置条件
   - 页面加载完成
2. 操作流程
   1) 用户输入文本（支持多行）
   2) 非空校验
   3) 点击发送
   4) 展示用户消息气泡
   5) 清空输入框
   6) 展示 AI loading 状态
   7) 调用优化 API
   8) 返回优化结果
3. 异常处理
   - 空输入：发送按钮禁用
   - 请求失败：显示错误提示 + 支持重试
   - 请求超时：显示超时提示
4. 性能要求
   - loading 反馈 < 200ms
   - API 响应 ≤ 5s（P50）

**验收标准**
- 输入非空文本后可发送
- 发送后可见用户消息与 loading
- 成功返回后展示 AI 结果
- 失败时可点击重试

**技术实现概要**
- 前端状态：`messages[]`、`input`、`isLoading`、`error`
- 后端接口：`POST /api/optimize`

### 阶段三：AI 对话生成

#### US-03：作为用户，我希望系统返回结构化且高质量的优化 Prompt，以便我可以直接复制使用

**价值陈述**
- 作为用户
- 我希望收到更清晰、更具体、可直接用于模型调用的 Prompt
- 以便提升 AI 结果稳定性与可控性

**业务规则与逻辑**
1. 输入处理规则
   - 对用户原始输入进行语义提炼
   - 补全关键缺失信息（目标、风格、约束、输出格式）
   - 保留用户核心意图，不改变任务本质
2. 输出结构规则（默认模板）
   - 任务目标（Goal）
   - 背景信息（Context）
   - 约束条件（Constraints）
   - 输出要求（Output Format）
   - 可选示例（Examples）
3. 内容安全与合规
   - 拒绝违法/危险内容优化请求
   - 对边界请求返回安全提示
4. 响应策略
   - 优先返回“可直接复制”的完整 Prompt
   - 若信息不足，先给出“优化草案 + 追问建议”

**验收标准**
- 返回内容可直接复制并用于第三方模型
- 返回内容包含至少 3 个结构化要素（如目标/约束/格式）
- 危险内容请求被拒绝并返回明确提示

**技术实现概要**
- LLM：DeepSeek（通过服务端 API 调用）
- Prompt 模板：系统提示词 + 用户原始输入
- 参数建议：`temperature` 0.3~0.7（可配置，默认 0.5）

---

### 阶段四：结果消费与调整

#### US-04：作为用户，我希望复制优化结果并继续迭代，以便逐步得到更符合预期的 Prompt

**价值陈述**
- 作为用户
- 我希望一键复制并补充新要求继续优化
- 以便快速完成多轮精修

**业务规则与逻辑**
1. 复制功能
   - 每条 AI 优化结果提供“复制”按钮
   - 成功复制后显示 toast：“已复制”
2. 多轮优化
   - 用户可继续输入“再简洁一点”“增加示例”等要求
   - 系统将新输入与历史上下文一并发送
3. 历史展示
   - 会话内展示消息历史
   - 刷新页面后历史清空（V1 不做持久化）

**验收标准**
- 点击复制后可在剪贴板获取完整文本
- 第二轮输入可基于上一轮结果继续优化
- 页面刷新后历史被清空

**技术实现概要**
- 前端：Clipboard API + 本地会话态（内存）
- 接口：同 `POST /api/optimize`，附带上下文 messages

## 5. 数据模型与接口契约

### 5.1 前端消息模型
```ts
interface Message {
  id: string;
  role: 'user' | 'assistant' | 'system';
  content: string;
  createdAt: string; // ISO 时间
  status?: 'sending' | 'success' | 'error';
}
```

### 5.2 API：优化接口
**Endpoint**：`POST /api/optimize`

**Request Body**
```json
{
  "messages": [
    { "role": "user", "content": "帮我写一个产品发布会主持词" }
  ],
  "options": {
    "temperature": 0.5,
    "maxTokens": 1200
  }
}
```

**Response 200**
```json
{
  "id": "opt_123",
  "output": "你是一位资深活动策划...",
  "meta": {
    "model": "deepseek-chat",
    "latencyMs": 1820
  }
}
```

**错误码约定**
- `400 INVALID_INPUT`：输入为空或格式错误
- `401 API_KEY_MISSING`：服务端未配置密钥
- `408 REQUEST_TIMEOUT`：上游模型超时
- `429 RATE_LIMITED`：请求频率超限
- `500 INTERNAL_ERROR`：服务内部错误
- `502 UPSTREAM_ERROR`：上游模型服务异常

## 6. 非功能需求（NFR）

### 6.1 性能
- 首屏加载 ≤ 2s（良好网络，桌面端）
- P50 响应 ≤ 5s，P95 ≤ 12s
- loading 可见反馈 < 200ms

### 6.2 可用性
- 错误信息可读、可重试
- 输入/发送/复制路径不超过 2 次点击

### 6.3 兼容性
- 支持最新版 Chrome/Edge/Safari
- 移动端最小宽度 360px 可用

### 6.4 安全与隐私
- 不做账号体系，不存储用户个人身份信息
- 默认不落库聊天内容（仅请求链路中转）
- 服务端密钥不下发前端

## 9. 风险与应对

- **风险1：上游模型波动导致响应慢/失败**
  - 应对：超时控制、重试机制、错误兜底提示
- **风险2：用户输入过于抽象，结果不稳定**
  - 应对：在响应中增加“补充信息建议”引导二次输入
- **风险3：V1 无存储导致用户刷新丢失上下文**
  - 应对：明确预期，并在后续版本评估本地草稿缓存
